# PDF Markdown 변환 및 청킹 방식

## 먼저 정정

이번 작업에서 청킹 자체는 LLM이 임의로 수행한 것이 아니라 `app/chunk_pdf.py`에 들어 있는 규칙 기반 로직으로 수행했습니다.

다만 최초 작업 당시에는 PDF를 별도 `.md` 파일로 저장해두지 않고, `pdfplumber`로 PDF 텍스트를 바로 추출해 `data/hackathon_chunks.json`을 만들었습니다. 과정을 확인할 수 있도록 원본 PDF에서 추출한 Markdown 기준 파일을 `data/hackathon_source.md`로 추가했습니다.

## 입력과 산출물

- 원본 PDF: `2026 성남시 정책 아이디어 청년 해커톤 대회 개최 공고.pdf`
- Markdown 추출본: `data/hackathon_source.md`
- 청킹 결과: `data/hackathon_chunks.json`
- 최종 청크 수: 11개

## Markdown 변환 방식

PDF 텍스트 추출은 `pdfplumber`로 했습니다.

페이지별로 아래 처리를 적용했습니다.

1. `page.extract_text(x_tolerance=1, y_tolerance=3)`로 텍스트 추출
2. 각 줄의 중복 공백을 하나로 정리
3. 빈 줄 제거
4. PDF 하단 페이지 번호 형태인 `- 1 -`, `- 2 -` 등 제거
5. 페이지 구분을 `## Page N` heading으로 저장

즉 `data/hackathon_source.md`는 사람이 보기 좋게 일부 정리한 텍스트 추출본이며, 원문 내용 자체를 요약하거나 LLM으로 재작성한 것은 아닙니다.

## 청킹 방식

청킹은 `app/chunk_pdf.py`의 `build_chunks()` 함수에서 수행합니다.

## 사용한 청킹 기법

사용한 기법은 **구조 인식 섹션 기반 청킹(Structure-aware Section Chunking)** 입니다.

이 방식은 문서를 단순히 일정 글자 수나 토큰 수로 자르는 것이 아니라, 문서 안에 있는 제목, 페이지, 붙임, 조항성 항목 같은 구조를 최대한 보존하면서 청크를 만드는 방식입니다.

이번 구현에서는 여기에 두 가지 보조 규칙을 더했습니다.

- **짧은 섹션 병합(Short-section Merge)**: 공고번호나 짧은 제목처럼 단독으로 의미가 부족한 조각은 인접 본문과 합침
- **긴 섹션 오버랩 분할(Overlapping Split)**: 너무 긴 섹션은 950자 기준으로 나누되, 앞 청크의 마지막 120자를 다음 청크에 겹쳐 문맥을 보존

따라서 전체 방식은 아래처럼 정리할 수 있습니다.

```text
PDF 페이지별 텍스트 추출
-> 페이지/제목 구조 유지
-> 숫자 제목 패턴으로 섹션 후보 분리
-> 너무 짧은 섹션은 병합
-> 너무 긴 섹션은 overlap을 두고 분할
-> page/section/source_path metadata 부여
```

## 이 기법을 사용한 이유

이 문서는 일반 산문이 아니라 공고문입니다. 공고문은 `신청기간`, `참가자격`, `제출서류`, `심사`, `시상`, `유의사항`, `붙임`처럼 구조가 답변 품질에 직접 영향을 줍니다.

그래서 고정 길이 청킹보다 구조 인식 섹션 기반 청킹이 더 적합하다고 판단했습니다.

선택 이유는 다음과 같습니다.

1. 출처 표시가 쉬움

   답변 화면에서 `p.1 > 성남시 공고...`, `p.4 > 5. 유의사항`처럼 근거 위치를 보여줘야 합니다. 페이지와 섹션을 유지하면 사용자가 근거를 바로 확인할 수 있습니다.

2. 의미 단위가 유지됨

   `참가자격`과 `신청기간`, `심사기준`, `시상금` 같은 정보가 중간에서 잘리면 RAG 답변이 누락되거나 엉뚱해질 수 있습니다. 섹션 단위로 묶으면 한 청크 안에 관련 조건이 함께 남습니다.

3. 문서가 짧고 구조가 명확함

   원본 PDF는 10쪽이고 최종 텍스트도 길지 않습니다. 500자 이하로 잘게 쪼개기보다, 페이지/섹션 중심으로 700-950자 정도의 청크를 만드는 편이 검색과 답변 생성 모두에 유리했습니다.

4. 리랭커가 판단할 맥락을 충분히 확보함

   Cohere rerank는 후보 문서 안의 문맥을 보고 질문과의 관련도를 다시 판단합니다. 너무 작은 청크보다, 제목과 본문이 함께 있는 청크가 리랭킹에 더 유리합니다.

5. 개인정보 동의서와 서약서 같은 붙임도 독립 근거로 보존 가능

   붙임 양식은 본문 공고와 성격이 다르므로 별도 청크로 유지했습니다. 덕분에 개인정보, 서약, 신청서 관련 질문에도 해당 붙임 청크가 근거로 잡힐 수 있습니다.

## 사용하지 않은 방식

이번 작업에서는 아래 방식은 사용하지 않았습니다.

- LLM 기반 의미 청킹: API 비용과 재현성 문제 때문에 사용하지 않음
- 고정 토큰 청킹: 공고문 제목/항목 구조가 끊길 수 있어 사용하지 않음
- 문장 단위 초미세 청킹: 청크 수가 불필요하게 많아지고 리랭킹 후보 품질이 낮아질 수 있어 사용하지 않음
- 전체 문서 1청크: 근거 위치 표시와 세부 질문 검색에 불리해 사용하지 않음

### 1. 페이지 단위 유지

먼저 PDF를 페이지별로 추출합니다. 각 청크의 metadata에 `page_start`, `page_end`를 넣기 위해 페이지 경계를 유지했습니다.

### 2. 제목 패턴으로 섹션 후보 분리

각 페이지 안에서 아래 정규식으로 섹션 후보를 나눴습니다.

```python
r"(?=^\d+\.\s+)"
```

예를 들면 아래 제목들이 분리 기준이 됩니다.

- `1. 개요`
- `2. 추진절차`
- `3. 심사 및 공개검증`
- `4. 시상`
- `5. 유의사항`

### 3. 너무 짧은 섹션 병합

처음에는 `성남시 공고 제2026-1704호`처럼 너무 짧은 청크가 생겼습니다. 이런 청크는 단독으로 검색될 때 의미가 부족하므로 `merge_short_sections()`로 인접 본문과 합쳤습니다.

기준값:

```python
min_chars = 260
max_chars = 1200
```

즉 260자보다 짧은 섹션은 가능하면 앞뒤 섹션과 합치고, 합친 결과가 1200자를 넘지 않도록 했습니다.

### 4. 긴 섹션 분할

섹션이 너무 길면 `split_long_text()`로 다시 나눴습니다.

기준값:

```python
max_chars = 950
overlap = 120
```

긴 본문을 나눌 때는 앞 청크의 마지막 120자를 다음 청크 앞에 붙여 문맥이 끊기지 않도록 했습니다.

### 5. 메타데이터 부여

각 청크에는 검색 결과 화면과 출처 표시를 위해 아래 metadata를 넣었습니다.

```json
{
  "document_key": "seongnam_youth_hackathon_2026",
  "document_title": "2026 성남시 정책 아이디어 청년 해커톤 대회 개최 공고",
  "source_type": "pdf",
  "page_start": 1,
  "page_end": 1,
  "section_title": "성남시 공고 제2026-1704호",
  "part": 1,
  "source_path": "2026 성남시 정책 아이디어 청년 해커톤 대회 개최 공고 > p.1 > 성남시 공고 제2026-1704호"
}
```

## 왜 11개 청크인가

이 PDF는 10쪽짜리 짧은 공고문이고, 대부분 페이지가 하나의 논리 단위입니다. 그래서 고정 토큰 수로 잘게 쪼개기보다 페이지와 제목 구조를 유지하는 방식이 출처 제시에 더 적합하다고 판단했습니다.

최종적으로 11개가 된 이유는 다음과 같습니다.

- 1쪽 공고 개요와 신청 정보: 1개
- 2쪽 추진절차와 1차 심사: 1개
- 3쪽 공개검증/최종평가/시상: 1개
- 4-5쪽 유의사항과 붙임 안내: 2개
- 6-10쪽 신청서/개인정보 동의서/서약서 붙임: 6개

## 임베딩과의 관계

Markdown 변환과 청킹은 API 없이 로컬에서 수행했습니다. API는 청킹 이후 단계에서 사용했습니다.

- 임베딩: Cohere `embed-v4.0`, 1536차원
- 리랭킹: Cohere `rerank-v3.5`
- 답변 생성: Cohere `command-r-08-2024`
